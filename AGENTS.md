# ops — Project Instructions

## Language

All content in this repository (skills, agents, scripts, docs, commit messages, PR descriptions) must be written in **English**. This includes code comments, markdown files, and any text generated by ops skills when working on this project.

## Stack-agnostic by default

All ops content (skills, agents, examples, templates, checklists) must remain **stack-agnostic**. The project targets any language, framework, architecture, deployment model, and runtime — not a specific stack.

Forbidden without explicit contextual justification:
- Hardcoded stack references in normative text: "Django setting", "env var", "JSONField", "serializer", "Rails", "Express middleware", "React hook", etc. If a concrete example is needed, introduce it as "e.g." and list at least two different-stack alternatives, or mark the example as illustrative.
- Architectural recommendations framed as universal truths: "server-driven is almost always cleaner", "the backend must own the rule", "permissions must live on the server". These assumptions break for offline-first apps, local-first apps, edge-computed apps, real-time collaborative apps, peer-to-peer systems, CLIs, and pure libraries.
- Feature-type assumptions: "the UI must be…", "the API endpoint should…". Not every feature has a UI or an API. Use generic terms like "interface surface" or "consumer".
- Stack-specific defaults in architectural choice templates (A/B/C questions). Present options neutrally; let the user's context drive the choice.

When an ops skill asks an architectural question, it must present the trade-offs and let the user decide — it must not pre-pick the answer based on an assumed stack. The one exception: hard security invariants (e.g. "never put secrets in client code") are stack-independent and stay normative.

When in doubt: if a reader from a non-web, non-Django, non-backend-authoritative context would feel the content doesn't apply to them, the content is too stack-specific and must be rewritten.

## Architecture: scripts vs. prompts

Logic with complex deterministic branching (config detection, baseline selection, structured parsing) lives in `bin/`. Scripts are prefixed `ops-` to avoid PATH namespace collisions.

Logic that requires judgment, contextual interpretation, or adaptation to unforeseen cases stays in skills markdown files.

When in doubt: if the logic can be expressed as a pure function of inputs (files, env vars, git state) with no need for LLM reasoning, it belongs in a script.

## Versioning

Every change must update the version in:
1. `.claude-plugin/plugin.json` (`version` field)
2. `.claude-plugin/marketplace.json` (`version` field)
3. `package.json` (`version` field)
4. `CHANGELOG.md` (new section at the top)

When adding or removing a skill, also update:
5. `data/bootstrap-context.md` (skill routing table)
